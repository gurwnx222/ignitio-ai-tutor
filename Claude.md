## Project Overview

**ignitio-ai-tutor** is an AI-powered LangChain learning assistant built with LangGraph. It takes a user's learning preferences and generates a personalised 7-day LangChain roadmap, then progressively delivers explanations and code examples **on-demand** (only when the user reaches a concept вЂ” never pre-generated).

The system follows an **orchestrator в†’ worker nodes в†’ aggregator** pipeline pattern.

## Graph State Schema

Defined in `graph/state.py` using Pydantic v2:

```python
class graph_state(BaseModel):
    user_query: str        # Raw input from the user
    roadmap: dict          # 7-day learning roadmap generated by the orchestrator
    sub_tasks: dict        # Daily sub-tasks broken down from the roadmap
    concept_map: dict      # All LangChain concepts mapped across the 7 days
    explaination: dict     # Concept explanations вЂ” generated lazily (day-of only)
    code_example: dict     # Code examples вЂ” generated lazily (day-of only)
    final_response: str    # Formatted output shown to the user вЂ” generated lazily
```

> **Important:** `explaination`, `code_example`, and `final_response` are **lazy** вЂ” they are only generated when the user actually reaches that day's concept, never pre-populated.

## Node Responsibilities

### Orchestrator (`nodes/orchestrator/`)
- Receives the raw `user_query`
- Generates a structured 7-day `roadmap` tailored to the user's preferences
- Breaks the roadmap into granular `sub_tasks` per day
- Builds the `concept_map` of all LangChain topics to be covered
- **Runs once** at the start of the session

### Explainer (`nodes/explainer/`)
- Receives the current day's concept from `concept_map`
- Populates `explaination[day]` with a clear, educational explanation
- **Runs on-demand** вЂ” only when the user progresses to that day

### Coder (`nodes/coder/`)
- Receives the current day's concept
- Populates `code_example[day]` with the most relevant, minimal code example
- **Runs on-demand** вЂ” only when the user progresses to that day

### Aggregator (implicitly via graph edges)
- Combines `explaination[day]` + `code_example[day]`
- Produces the clean `final_response` string for UI rendering
- **Runs on-demand** after both worker nodes complete for that day

## Key Conventions

- **Lazy generation:** Never pre-generate `explaination`, `code_example`, or `final_response` for future days. Only generate for the day the user has reached.
- **Prompt isolation:** Each node's prompt lives in its own `prompt.py` вЂ” keep business logic out of prompts and vice versa.
- **State immutability:** Treat `graph_state` as append-only where possible; avoid overwriting previously generated days.
- **LLM config centralised:** All LLM initialisation goes in `core/llm.py`. Nodes import from there вЂ” never instantiate models inline.
- **Environment variables:** All secrets and API keys go in `.env` and are loaded via `dotenv`. Never hardcode credentials.

## Development Notes for Claude Code

- When adding a **new node**, follow the `nodes/<name>/node.py` + `nodes/<name>/prompt.py` pattern
- When modifying **graph topology** (adding edges, changing flow), edit `graph/builder.py` and `graph/edges.py` together
- The `graph_state` schema in `graph/state.py` is the **single source of truth** for what data exists in the graph вЂ” always update it when adding new state fields
- LangGraph nodes should accept and return `graph_state` (or a partial dict compatible with it)
- Keep prompts in `prompt.py` files as plain strings or `PromptTemplate` objects вЂ” no logic
